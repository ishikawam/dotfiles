# CLAUDE.md - グローバル設定

このファイルはすべてのプロジェクトでClaude Code (claude.ai/code) に対するグローバルなガイダンスを提供します。

## 言語設定

**日本語での対応**
- すべてのやり取りを日本語で行う。特別に他の言語を要求されない限り日本語で回答する。技術的な内容も日本語で説明する。

## コミュニケーション

- 変更を加える前に、影響範囲と方針を提示し承認を得ること。承認なしにファイルを書き換えない。
- 寄り添いや同調はしない。私の主張が誤っていれば根拠を添えて明確に指摘すること。
- 不確実なことを断定しない。推測と事実を区別して述べる。
- 回答は簡潔に。前置き・要約の繰り返しは不要。
- トークンは惜しまない。冗長な出力ではなく、調査・検証・思考の深さに使うこと。

## 作業の進め方

- 不明点は推測で進めず、先に質問する。ただし質問は要点を絞って一度にまとめる。
- 大きなタスクは着手前に計画（ステップ・影響ファイル・検証方法）を提示する。
- 変更後はテストと型チェック・lint の実行結果を必ず報告する。失敗を黙って残さない。
- 私の指示に技術的な問題があると判断したら、従う前に指摘すること。

## 設計原則

- セマンティックな命名・構造を優先。型、関数名、モジュール境界が意図を語ること。
- DRY と YAGNI を厳守。使われない抽象、将来用の引数、デッドコードを作らない。
- DB スキーマは原則として正規化する（第3正規形以上）。非正規化は計測された理由がある場合のみ、コメントで根拠を残して行う。
- 関数は単一責任。副作用と純粋ロジックを分離する。
- 「とりあえず動く」実装、その場しのぎのパッチ、TODO の置き土産を禁止。根本原因に対処する。
- 最新の安定したアーキテクチャ・ライブラリを優先するが、流行のみを理由に採用しない。トレードオフを述べてから選ぶこと。

## エラーハンドリング

- catch-all を書かない。想定する例外型のみを個別にハンドルする。
- 例外を握り潰さない。ログだけ出して継続するような実装は禁止。
- 暗黙のフォールバック（null 返却、デフォルト値での誤魔化し、サイレントリトライ等）を量産しない。フォールバックを入れる場合は意図と条件をコメントで明示。
- 「想定外」は早期に失敗させる（fail-fast）。

## コード品質原則

- public な関数・型・モジュールには意図と契約を説明するコメント（なぜ、何を保証するか）を残す。実装をなぞるだけのコメントは書かない。
- マジックナンバー・マジックストリングは定数化または列挙型化。
- 既存コードの規約・命名・レイヤリングを尊重し、勝手にスタイルを変えない。

## リファクタリング

- タスクに関連する範囲で、明らかな改善点があれば提案する（自動実行はしない）。
- 大規模な変更は段階に分け、各段階で動作確認できる状態を保つ。

## ドキュメント管理原則

- コード変更に伴うドキュメント（README、ADR、API ドキュメント、コメント）の更新漏れを指摘し、必要なら更新する。
- ドキュメントには現状の設計・仕様のみを記す。変更経緯・履歴・「以前は〜だった」等は書かない（git log の役割）。
- ADR は意思決定の根拠と代替案、トレードオフを簡潔に。

## 重要なルール

**明示的な要求なしにコードをコミットしない**
- ユーザーが明示的にコミットを要求しない限り、`git commit` を実行しない。
- ユーザーが「git commit」と言った場合は、即座にコミットを実行する。
- このルールは絶対に遵守し、常に従わなければならない。

**GITコミットポリシー - 重要ルール**
- **`git add` を実行してはいけない - 全プロジェクト共通**。`git add -A` / `git add .` / `git add -u` 等あらゆる形式を含む。ユーザーがコミットを要求しても、関連ファイルであっても自動的にステージングしない。
- ユーザーが「git commit」と言った時は、`git commit` コマンドのみを実行する。ユーザーが既にステージングしたファイルのみをコミットする。
- ステージング済みと未ステージングの変更が混在している場合、それはユーザーの意図的な選択と判断し、確認せずステージング済みのみでコミットを実行する。
- ステージングされたファイルがない場合は、コミットする変更がないことをユーザーに伝える。
- これは交渉の余地のないルール。
- **`git commit` 実行前は必ず `git status` で確認する**。コミット可能な変更があるかを事前に確認する。
- **`git status --short` と `git status --porcelain` の使用を禁止する**。1列目と2列目の `M` / ` M` を誤読するリスクが高い。常に `git status` の通常出力を使う。
- **コミットメッセージ作成時の必須手順**:
  - 必ず `git diff --cached` でステージングされた変更の詳細内容を確認する。
  - ブランチ名やファイル名からの推測ではなく、実際の変更内容に基づいてメッセージを作成する。
  - ステージングされていない変更内容や、実際の変更と一致しないメッセージを作成しない。

**GitHub PR/Issueポリシー - 必須ルール**
- PR / Issue / コメントの作成・編集時は本文末尾に必ず以下の署名を含める。例外なし。
  ```
  🤖 Generated with [Claude Code](https://claude.ai/code)
  ```
- 対象コマンド: `gh pr create` / `gh pr edit` / `gh issue create` / `gh issue edit` / `gh pr comment` / `gh issue comment`。
- 目的: AI アシスタントが作成したコンテンツであることを明確にする。

**ファイル作成ポリシー**
- 要求されたことを行う。それ以上でも以下でもない。
- 目的達成に必要でない限りファイルを作成しない。新規作成より既存ファイルの編集を優先する。
- ドキュメントファイルはユーザーが明示的に要求した場合のみ作成する。

**プロジェクト初期化ポリシー**
- **新しいプロジェクトの開始時またはユーザーが `init` コマンドを実行した時は以下のファイルを生成する**:
  - `.editorconfig`（コード書式統一）
  - `README.md`（プロジェクト説明とセットアップ手順）
  - `.gitignore`（プロジェクトタイプに応じた適切な除外ルール）
  - `Makefile`（依存関係管理とビルドコマンド、必要に応じて）
- **自動生成の条件**: 「init」「初期化」「setup」「セットアップ」「プロジェクト作成」等のキーワード、新規プロジェクトディレクトリでの作業開始時、ユーザーが明示的に初期ファイル作成を要求した時。

**gitignoreポリシー**
- **必要最小限の除外ルールのみを記載する**。プロジェクトに実際に存在するファイル/ディレクトリのみを除外対象とし、汎用的すぎるルールや使用しない技術のルールは含めない。
- **パスの使い分け**:
  - **絶対パス（`/` 始まり）**: ルート固有のファイル（例: `/.env`, `/vendor`, `/node_modules`, `/public/build`）
  - **相対パス**: どのディレクトリでも発生しうる OS ファイル（例: `.DS_Store`, `Thumbs.db`）
- カテゴリごとにコメントで整理する。

**.editorconfigポリシー**
- 作成した .editorconfig を厳格に遵守する。ファイル作成・編集前に必ず設定を確認し、それに従う。
- 適切なインデント（スペース 2 文字、Makefile はタブ）、UTF-8 エンコーディング、LF 改行コード、末尾改行（`insert_final_newline = true`）、末尾空白の除去（`trim_trailing_whitespace = true`、`*.md` は除外）を守る。

**セキュリティポリシー**
- 秘密情報やキーをコードに露出・ログ出力・リポジトリにコミットしない。
- **Git トラッキング状況の必須確認**:
  - ファイル削除・変更前に `git ls-files` と `git status` で追跡状況を確認する。
  - Git で追跡されていないファイル（個人の `.env` 含む）は明示的に要求されない限り変更しない。
- **プライベート/パブリックの区別**: Git 追跡ファイル（変更可能）とユーザーのプライベートファイル（許可なしに触らない）を常に区別する。
- **機密情報の保護**:
  - コードや追跡ファイルに個人情報（URL、名前、トークン）をハードコードしない。
  - テンプレートファイル（`.env.example`、`.sample.md`）のみ変更可能。
  - 機密性の高い個人情報・業績データを含むファイルは慎重に扱い、内容を外部に漏洩させない。分析やレポート作成時も個人情報に配慮する。
- **プライベートディレクトリの保護**: ダウンロードディレクトリやユーザー作成ディレクトリは個人データを含むため、明示的な許可なしに変更しない。

**コードスタイル**
- **既存のコメントを勝手に削除しない**。変更対象外のコメントは絶対に触らない。内容が変更後のコードと矛盾する場合は削除ではなく修正する。削除はユーザーの明示的な指示がある場合のみ。
- ファイル変更時はまずそのファイルのコード規約を理解し、既存のライブラリ・ユーティリティ・パターンに従う。
- **ソースコードのコメントはすべて日本語で書く**。
- **例外**: 技術的な固有名詞・サービス名は英語のまま使用する（例: `# TKE Managed Cluster`, `# NAT Gateway`, `# AWS Lambda`）。理由は公式ドキュメントやコンソールと一致させるため。

**コメント記述ポリシー - 絶対ルール**
- 技術的事実のみを簡潔に記述する。無駄・意味不明なコメント、冗談・比喩・感情的表現は残さない。自己説明的なコードを書き、コメントは必要最小限にする。
- **禁止される表現**: 過激な表現（「〜を駆逐」「〜を撲滅」等）、曖昧な表現（「〜かもしれない」「〜だろう」等）、ユーモアや皮肉、過去の経緯や削除理由の説明。
- **許可されるコメントの例**: 技術的制約の説明（`# ECS Exec経由でRDS接続`）、設定値の説明（`# RDS接続用の変数`）、セクション区切り（`# terraform`, `# release`）。

**ドキュメント・コメント・ヘルプ記述ポリシー - 絶対ルール**
- **括弧書きの補足説明を入れない**。Makefile のヘルプ、コミットメッセージ、ドキュメントなど全てに適用。必要最小限の情報のみ記述する。
- **例**:
  - 悪い: `path: ## $PATHを見やすく表示（先頭2スラッシュでグループ化）` / `Add user authentication (using JWT tokens)` / `データベース接続を追加（PostgreSQL使用）`
  - 良い: `path: ## $PATHを見やすく表示` / `Add user authentication` / `データベース接続を追加`
- 実装の詳細や方法はコードを見れば分かるため記述不要。

**人名表記ポリシー - 絶対ルール**
- 漢字表記が確実にわかる場合のみ漢字を使用する。不明な場合は推測せずアルファベット表記（ローマ字）を使う。
- **例**:
  - 悪い: 「石川正幸」（`Masayuki Ishikawa` とのみ記載の場合）
  - 良い: 「Masayuki Ishikawa」（漢字が不明）/「石川将行」（漢字が確認済み）

**依存関係管理ポリシー - 重要ルール**
- **brew で依存関係をインストールする前に、必ず Makefile を作成する**。`make install` ターゲットを含め、他の開発者が同じ環境を再現できるようにする。
- 外部ツール（`gh`、`jq` 等）も `make install` で一括インストールできるようにする。
- **絶対にグローバル環境にプログラミング言語パッケージをインストールしない**。一時的な作業や単発ツールも含め、すべてのセッションで以下のローカル環境管理を使う:
  - Python: pyenv（`.python-version` + `requirements.txt`）
  - Node.js: nodenv（`.node-version` + `package.json`）
  - Ruby: rbenv（`.ruby-version` + `Gemfile`）
  - Go: goenv（`.go-version` + `go.mod`）
  - Java: jenv（`.java-version`）
  - Rust: rustup（`rust-toolchain.toml`）

**GitHub CLI認証ポリシー**
- GitHub API を使うプロジェクトではセットアップ手順に `gh auth login` を含め、認証が必要なコマンドの実行前に認証状態を確認する。認証エラー時はユーザーに `gh auth login` の実行を案内する。

**Terraformプロジェクトポリシー**
- **初期化・フォーマット**: `terraform init` は `make install` または `make install-base` で、フォーマットは `make fmt` で実行する。手動で terraform コマンドを直接実行せず、Makefile で定義されたコマンドを使う。
- **AWS 認証とプロファイル管理**: 認証は `make login` を使う。プロジェクト固有の AWS プロファイルを尊重する。IAM Identity Center 使用時は適切な SSO ログインコマンドを使う。
- **セキュリティ**: 本番環境への変更は特に慎重に実行し、事前確認を徹底する。
- **環境別設定管理**:
  - 環境ごとに独立したディレクトリ構成（`env_dev/`, `env_prod/` 等）。
  - 共通モジュールは `common/` や `modules/` で管理。
  - 環境固有の変数は `terraform.tfvars` で管理。
  - 各環境は独立して初期化し、相互に影響しないようにする。

**AWS Session Managerポリシー**
- **SSH/RDS アクセスは Session Manager を優先**する。SSH ポート（22）を開放不要、CloudTrail にアクセスログが残り監査性が向上する。
- **標準コマンド**:
  - `make ssm-port-forward` — RDS へのポートフォワーディング
  - `make ssm-connect` — EC2 へのシェルアクセス
  - `make ssm-list-instances` — 利用可能なインスタンス一覧表示
- **前提条件**: AWS CLI v2、Session Manager Plugin（`brew install --cask session-manager-plugin`）、EC2 への SSM Agent インストール、IAM への適切な SSM 権限。

**プロジェクト開発準備ポリシー**
- **新しいプロジェクト開発時**: 開発環境構築を必須化する。
- **API 連携プロジェクト**: 設計確認事項を明確にし、環境変数管理（.env テンプレート作成と API 認証情報の安全管理）を必須化する。定期的なデータ取得・分析処理が必要な場合はバッチ処理システムを設計に含める。

**ドキュメント連動管理ポリシー**
- 提案書・見積書・仕様書・構成図など関連ドキュメント群は整合性を維持する。1 つを更新する際は関連ファイルへの影響を確認し、金額・工数・技術スタック・システム構成の一貫性を保つ。
- **更新時の必須チェック項目**:
  - 金額・工数: 提案書と詳細見積書の整合性
  - 技術スタック: API 仕様書と提案書の技術選定
  - システム構成: 構成図と各ドキュメントのアーキテクチャ記述
  - プロジェクトスコープ: 要件定義と提案書の機能範囲

**依頼元ドキュメント保護ポリシー**
- 依頼元・クライアント提供の原本ドキュメント（契約書類、要件定義書、仕様書など）は絶対に変更・削除しない。参照のみ可能。改変・削除は契約違反となりうるため厳禁。
- 原本ディレクトリ（例: `docs/from_client/`）は編集対象外として明記する。

**Excel/CSV管理ポリシー**
- Excel ファイルは必ず CSV 形式に変換してから AI 処理を行う。専用の変換ツール（`excel-converter` 等）を使い、手動変換は避ける。`make convert` のような自動化コマンドで変換作業を効率化する。
- 分析やレポート作成時は個人情報・機密データに特に配慮する。

**Dockerコンテナ管理ポリシー**
- docker-compose を使うプロジェクトでは Makefile で以下のコマンドを統一する:
  - `make up` — すべてのサービスを docker-compose で起動
  - `make down` — すべてのサービスを停止
  - `make destroy` — コンテナ、ボリューム、ログをクリーンアップ

**テスト実行ポリシー**
- **標準コマンド**:
  - `make test` — API テストをカバレッジ付きで実行
  - `make unittest` — ユニットテストを実行
  - `make coverage` — カバレッジレポートを生成し HTML レポートを開く
- **個別テスト**: Docker コンテナ内での実行も考慮する（例: `docker-compose exec [service] go test -v ./[package-path]`）。
- **テスト品質**: ユニット / 統合 / E2E テストを適切に分離する。テストデータはテスト専用のものを使い、本番データを使わない。

**コード品質管理ポリシー**
- **標準コマンド**: `make lint`（リンティング、言語別の標準ツール使用） / `make fmt`（コードフォーマット） / `make check`（静的解析、`staticcheck` 等）。
- **Go 言語**: `gofmt`、`goreturns`、`golint`、`golangci-lint` を活用。Docker コンテナ内でのリンティングも考慮する。

**Go言語開発ポリシー**
- **標準プロジェクト構成**:
  - `cmd/` — 実行可能ファイル（`server/`, `batch/`, `worker/` 等）
  - `handlers/` — HTTP リクエストハンドラ
  - `models/` — データモデル
  - `service/` — ビジネスロジック
  - `middleware/` — HTTP ミドルウェア
  - `routers/` — ルート定義
  - `pkg/` — 共有パッケージ
  - `resources/` — 静的アセット、テンプレート
  - `settings/` — 設定管理
- **環境設定**: 設定ファイルは `env/` ディレクトリで環境別管理、`cp env/local.ini app.ini` パターンでローカル設定をコピー、本番用とテスト用を分離する。

**Laravel/PHP開発ポリシー**
- **標準コマンド**: `make setup`（初期セットアップ） / `make install`（依存関係・環境設定・キャッシュクリア） / `make migrate` / `make dev` / `make up` / `make down` / `make exec`（PHP コンテナ SSH） / `make log`（Laravel ログ tail）。
- **コード品質**: `make fix`（Pint、PSR-12 自動整形） / `make analyse`（PHPStan レベル 5） / `make test-phpstan` / `make generate-phpstan-baseline`（ベースライン更新） / `make test-pint`（スタイルチェック） / `make fix-pint`（自動修正）。
- **テスト実行**: `docker compose exec php ./vendor/bin/phpunit` を基本形とし、対象を `tests/Unit` / `tests/Feature` / 特定ファイル / `--filter testExample` で指定する。
- **API 文書化**: `make create-swagger` で生成、Swagger UI は `http://localhost:8081/api/documentation`。
- **開発規約**: Laravel Pint（PSR-12 準拠、全 PHP ファイル自動整形）、PHPStan（型安全性と品質チェック）、Editorconfig（インデント 4 スペース、UTF-8、LF）。
- **必須のコンテナ実行**: すべての Laravel / PHP / Composer コマンドは **必ず Docker コンテナ内** で実行する。正: `docker compose exec php php artisan migrate`, `docker compose run php composer install`, `make exec`。避ける: ホスト直接実行の `php artisan migrate` や `composer install`。
- **ログ確認**: `make log`（リアルタイム監視） / `docker compose exec php tail -f storage/logs/laravel.log`（特定ファイル）。ログは `storage/logs/` に保存。

**環境設定管理ポリシー**
- **複数環境対応**: 環境別設定ファイル（local, develop, staging, production 等）を適切に分離する。環境変数は .env で管理し機密情報は保護する。API エンドポイントは環境に応じて自動切り替えする。本番と開発で異なる設定が必要な項目を明確にする。

**静的サイト開発ポリシー（AWS S3/CloudFront）**
- **標準構成**: S3 静的サイトホスティング + CloudFront CDN 配信、Route 53 ドメイン管理（オプション）、CloudFront 経由の HTTPS 自動対応。

**シェルコマンドポリシー - 絶対ルール**
- **`rm` は必ず `rm -f` または `rm -rf` を使う**。ユーザー環境では `rm` が `rm -i` にエイリアスされているため、単体使用は対話プロンプトで Bash ツールがハングする。これは交渉の余地のないルール。

**制限ファイル**
- `~/.config/git/ignore` は変更・編集しない。常に手を付けずにそのままにする。
